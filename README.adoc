== Quick start

* Validate template: `make validate`
* Build image: `make build`
* Run VM + role disks: `make run`

== Initialization policy (intentional)

This project intentionally keeps VM initialization user-driven (interactive + debug by default).

Reason: the Packer boot/command driver flow for this macOS setup is currently fragile, buggy in practice, and costly to maintain over time.
Using interactive initialization keeps the workflow predictable and easier to recover when macOS UI behavior changes.

== Required VM setting

During initial VM setup, always enable Remote Login in macOS Settings (at least for your user):

* `System Settings -> General -> Sharing -> Remote Login`

Remote Login is required for reliable SSH-based provisioning/automation after first boot.

Provisioning credentials must match the template SSH settings:

* Username (bootstrap during provisioning):
** clone mode: `admin`
** ipsw mode: value of `macos_primary_account_name` (via Make: `.account.primary-name`)
* Password: `admin`

NOTE: Account alias login over SSH is currently not working for this image flow.
For example, even when `nxmatic` is configured as an alias, SSH login must still use the
primary macOS short name (`admin`).

If you change the macOS account password during setup, update `ssh_password` in `templates/vanilla-tahoe.pkr.hcl` accordingly.

== Display access choice

For interactive setup, prefer Tart-handled display/VNC access.

* Recommended default: use Tart display access + guest `Remote Login` (SSH).
* Optional: enable macOS `Screen Sharing` only when you explicitly need guest-provided remote GUI access.

In short: `Remote Login` is required; macOS `Screen Sharing` is optional.

== Defaults

* VM name: `nxmatic-macos`
* Display geometry: `1728x1080`
* Root disk: `100G`
* Role disks (ASIF):
** User Data: `160G`
** User Library: `40G`
** Git Bare Store: `8G`
** Git Worktree Store: `9G`
** Nix Store: `180G`
** Build Chains: `64G`
** VM Images: `512G`
* Initial in-VM APFS sizes:
** User Data: `64G`
** User Library: `20G`
** Git Bare Store: `4G`
** Git Worktree Store: `6G`
** Nix Store: `90G`
** Build Chains: `16G`
** VM Images: `120G`

Disk paths default under:

`~/.tart/disks/<vm_name>/`

== Sizing policy

For role disks, recommended max sizing policy is:

* `max_size_gb = ceil(current_used_gb Ă— 3)`

This corresponds to 100% current usage + 200% growth headroom.

The `Build Chains` disk is intended for toolchain-heavy paths such as `~/go`, `~/.m2`, `~/.npm`, and `~/.cache` to reduce pressure on the root/system container.

== Git Store layout (bare + worktrees)

Git data now uses dedicated role disks and a split model:

* Git Bare Store disk (mounted to host path in VM): `/private/var/lib/git/bare`
* Git Worktree Store disk (mounted to host path in VM): `/private/var/lib/git/worktrees`

Within that model:

* bare repositories live under `/private/var/lib/git/bare`
* working trees live under `/private/var/lib/git/worktrees`

During provisioning, `scripts/setup-git-store-layout.sh` can migrate existing repositories from legacy locations (for example `~/Git Store` or flat layouts under `/private/var/lib/git`) into the split model.

Relevant `.env` knobs (generated by `make env`):

* `GIT_STORE_LAYOUT_MODE=split|flat` (default: `split`)
* `GIT_STORE_MIGRATE_EXISTING=1|0` (default: `1`)
* `GIT_STORE_BARE_ROOT` (default: `/private/var/lib/git/bare`)
* `GIT_STORE_WORKTREE_ROOT` (default: `/private/var/lib/git/worktrees`)

== Override example

`make build .vm-name=bioskop-macos .tart.disk.root.max-size-gb=120 .tart.disk.user-data.max-size-gb=240 .tart.disk.user-data.initial-size-gb=96 .tart.disk.nix-store.initial-size-gb=120`

== VSCode-focused background service trimming

During provisioning, `scripts/trim-vscode-vm-services.sh` applies a lightweight background-service profile intended for SSH/VSCode-heavy VM usage.

Default behavior:

* Disable Spotlight indexing across mounted APFS volumes.
* Add `.metadata_never_index` markers on mounted APFS volumes.
* Purge Spotlight assets by default using `mdutil -E` on mounted APFS volumes.
* Fallback to direct `.Spotlight-V100*` cleanup only when `mdutil -E` fails on a writable mount.
* Disable Spotlight daemon:
** `com.apple.metadata.mds`
* Disable media analysis daemons:
** `com.apple.mediaanalysisd`
** `com.apple.photoanalysisd`
* Disable Siri/suggestions style user agents for the primary user:
** `com.apple.assistantd`
** `com.apple.parsecd`
** `com.apple.suggestd`
** `com.apple.knowledge-agent`
* Keep continuity/nearby-sharing enabled by default (safer for multi-host workflows).
* Optionally disable continuity/nearby-sharing style user agents for the primary user when `DISABLE_AIRDROP_CONTINUITY=1`:
** `com.apple.sharingd`
** `com.apple.coreservices.useractivityd`

Safety notes:

* Core SSH/network services are intentionally not disabled.
* Some launchd labels can vary by macOS release; disabling is best-effort.

Control toggles:

* `DISABLE_SPOTLIGHT_INDEXING` (default `1`)
* `SPOTLIGHT_DISABLE_MDS_DAEMON` (default `1`)
* `SPOTLIGHT_PURGE_ASSETS` (default `1`)
* `DISABLE_MEDIA_ANALYSIS_DAEMONS` (default `1`)
* `DISABLE_SIRI_AND_SUGGESTIONS` (default `1`)
* `DISABLE_AIRDROP_CONTINUITY` (default `0`)

Service mode:

* `SERVICE_TRIM_MODE=disable|enable` (default `disable`)

Re-activation examples inside VM:

* Re-enable all service groups currently selected by toggles:
** `sudo env SERVICE_TRIM_MODE=enable /usr/local/sbin/trim-vscode-vm-services`
* Re-enable continuity services only (when they were explicitly disabled):
** `sudo env SERVICE_TRIM_MODE=enable DISABLE_SPOTLIGHT_INDEXING=0 DISABLE_MEDIA_ANALYSIS_DAEMONS=0 DISABLE_SIRI_AND_SUGGESTIONS=0 DISABLE_AIRDROP_CONTINUITY=1 /usr/local/sbin/trim-vscode-vm-services`
