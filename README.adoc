= nxmatic macOS image build

Minimal, owner-focused Tart/Packer project to build a customized Tahoe vanilla image.

== Layout

* `templates/vanilla-tahoe.pkr.hcl` — vanilla/grow-only Packer template
* `templates/nxmatic-tahoe.pkr.hcl` — nxmatic customization Packer template
* `scripts/` — provisioning scripts
* `data/` — launchd plist payloads
* `docs/runbooks/disk-image-shrink.adoc` — migration runbook for shrinking host disk images on existing VMs
* `.flox/env/manifest.toml` — tool environment (`packer`, `tart`)
* `Makefile` — automated commands

== Runbooks

* xref:docs/runbooks/disk-image-shrink.adoc[Host disk-image shrink migration]

== Quick start

* Validate template: `make validate`
* Build image: `make build`
* Run VM + role disks: `make run`

== Recovery / console cheatsheet

Use these as the canonical operator commands when reconnecting to this project.

* Recovery boot with VNC + split role disks (recommended):
** `make run-recovery-console .tart.vm-name=nikopol .tart.run.disk.opts=sync=none,caching=cached`
* Normal boot with VNC + split role disks:
** `make run-console .tart.vm-name=nikopol .tart.run.disk.opts=sync=none,caching=cached`
* Rebuild image (interactive/debug defaults):
** `make build`
* Show exact generated command without running it:
** `make -n run-recovery-console .tart.vm-name=nikopol .tart.run.disk.opts=sync=none,caching=cached`

Notes:

* The `run-console` and `run-recovery-console` targets already attach split Git disks (`git-bare-store` + `git-worktree-store`) and `vm-images`.
* Keep `--recovery` on the same Tart command line (the Make target handles this).
* Role disk images live under `~/.tart/disks/<vm_name>/` by default and are reused when present.

== Initialization policy (intentional)

This project intentionally keeps VM initialization user-driven (interactive + debug by default).

Reason: the Packer boot/command driver flow for this macOS setup is currently fragile, buggy in practice, and costly to maintain over time.
Using interactive initialization keeps the workflow predictable and easier to recover when macOS UI behavior changes.

== Required VM setting

During initial VM setup, always enable Remote Login in macOS Settings (at least for your user):

* `System Settings -> General -> Sharing -> Remote Login`

Remote Login is required for reliable SSH-based provisioning/automation after first boot.

Provisioning credentials must match the template SSH settings:

* Username (bootstrap during provisioning):
** clone mode: `admin`
** ipsw mode: value of `macos_primary_account_name` (via Make: `.account.primary-name`)
* Password: `admin`

NOTE: Account alias login over SSH is currently not working for this image flow.
For example, even when `nxmatic` is configured as an alias, SSH login must still use the
primary macOS short name (`admin`).

== Login troubleshooting (GUI + CLI)

If the login screen shows user tiles (no username field), click the
`Stephane Lacoin (aka nxmatic)` tile and enter the password.

If the login screen is in name/password mode (username field visible), use the
short account name (`nxmatic`) as the username — not the full display name.

If the primary account appears locked after repeated failed attempts, unlock/reset from CLI:

* In guest VM terminal (or via `tart exec`):
** `sudo pwpolicy -u nxmatic -setpolicy "isDisabled=0"`
** `sudo dscl . -create /Users/nxmatic failedLoginCount 0`
** `sudo pwpolicy -u nxmatic -authentication-allowed`

This repository's provisioning sequence now includes `scripts/unlock-primary-account.sh`
to ensure the primary account starts in an unlocked state.

If you change the macOS account password during setup, update `ssh_password` in `templates/vanilla-tahoe.pkr.hcl` accordingly.

== Display access choice

For interactive setup, prefer Tart-handled display/VNC access.

* Recommended default: use Tart display access + guest `Remote Login` (SSH).
* Optional: enable macOS `Screen Sharing` only when you explicitly need guest-provided remote GUI access.

In short: `Remote Login` is required; macOS `Screen Sharing` is optional.

== Defaults

* VM name: `nikopol`
* Display geometry: `1728x1080`
* Root disk: `100G`
* Role disks (ASIF):
** User Data: `160G`
** User Library: `40G`
** Git Bare Store: `8G`
** Git Worktree Store: `9G`
** Nix Store: `180G`
** Build Chains: `64G`
** VM Images: `512G`
* Initial in-VM APFS sizes:
** User Data: `64G`
** User Library: `20G`
** Git Bare Store: `4G`
** Git Worktree Store: `6G`
** Nix Store: `90G`
** Build Chains: `16G`
** VM Images: `120G`

Disk paths default under:

`~/.tart/disks/<vm_name>/`

== Sizing policy

For role disks, recommended max sizing policy is:

* `max_size_gb = ceil(current_used_gb × 3)`

This corresponds to 100% current usage + 200% growth headroom.

The `Build Chains` disk is intended for toolchain-heavy paths such as `~/go`, `~/.m2`, `~/.npm`, and `~/.cache` to reduce pressure on the root/system container.

== Git Store layout (bare + worktrees)

Git data now uses dedicated role disks and a split model:

* Git Bare Store disk (mounted to host path in VM): `/private/var/lib/git/bare`
* Git Worktree Store disk (mounted to host path in VM): `/private/var/lib/git/worktrees`

Within that model:

* bare repositories live under `/private/var/lib/git/bare`
* working trees live under `/private/var/lib/git/worktrees`

During provisioning, `scripts/setup-git-store-layout.sh` can migrate existing repositories from legacy locations (for example `~/Git Store` or flat layouts under `/private/var/lib/git`) into the split model.

Relevant `.env` knobs (generated by `make env`):

* `GIT_STORE_LAYOUT_MODE=split|flat` (default: `split`)
* `GIT_STORE_MIGRATE_EXISTING=1|0` (default: `1`)
* `GIT_STORE_BARE_ROOT` (default: `/private/var/lib/git/bare`)
* `GIT_STORE_WORKTREE_ROOT` (default: `/private/var/lib/git/worktrees`)

== Override example

`make build .vm-name=bioskop-macos .tart.disk.root.max-size-gb=120 .tart.disk.user-data.max-size-gb=240 .tart.disk.user-data.initial-size-gb=96 .tart.disk.nix-store.initial-size-gb=120`

== Working VM + foreign disk source defaults

This project uses a single canonical working VM name plus explicit foreign disk-source controls:

* Working VM: `nikopol`
* Foreign Tart root: `.tart.foreign`
* Renamed-source VM (for auto attach): `nikopol`

You can set them in `.env`:

* `TART_WORKING_VM=nikopol`
* `TART_FOREIGN_HOME=.tart.foreign`
* `TART_EXTRA_DISKS_RENAMED_FROM_VM=nikopol`

`Makefile` reads these values at parse time, so changing `.env` adjusts the default
working VM and foreign renamed-disk source behavior used by targets.

== Incremental disk transition (mount legacy disks alongside new ones)

You can mount replicated legacy disk images (for example from `nikopol`) alongside
the current role disks while you migrate incrementally.

Use `.tart.run.extra-disks` with a space-separated list of image paths:

* Example:
** `make run-console .tart.run.extra-disks="$(HOME)/.tart/disks/nikopol/user-data.asif $(HOME)/.tart/disks/nikopol/git-store.asif"`

Or provide only a VM name and let Make discover `*.asif` files automatically:

* Example:
** `make run-console .tart.run.extra-disks.from-vm=nikopol`

When using `.tart.run.extra-disks.from-vm`, role-disk duplicates are filtered by default
based on filename (`user-data.asif`, `git-store.asif`, etc.) so canonical role disks for
the current VM are not attached twice.

If you intentionally want duplicate role attachments, enable:

* `make run-console .tart.run.extra-disks.from-vm=nikopol .tart.run.extra-disks.allow-duplicate-roles=1`

By default, extra disks use `.tart.run.extra-disk.opts=sync=none`.
Override if needed:

* `make run-console .tart.run.extra-disks="..." .tart.run.extra-disk.opts="sync=none,caching=cached"`

This keeps current VM role disks attached while making legacy volumes visible for copy/verification.

== Use an external disk as a canonical role

Yes — you can map a role directly to an external image path by overriding the role disk variable.

Example (use `nikopol` Git Store as the current VM Git Store role):

* `make run-console .tart.disk.git-store.image-path="$(HOME)/.tart/disks/nikopol/git-store.asif"`

Useful role overrides:

* `.tart.disk.user.image-path`
* `.tart.disk.user-library.image-path`
* `.tart.disk.git-bare-store.image-path`
* `.tart.disk.git-store.image-path`
* `.tart.disk.nix-store.image-path`
* `.tart.disk.user-build-chains-cache.image-path`
* `.tart.disk.vm-images.image-path`

Note: role naming in this flow is canonical; at runtime Tart attaches disk paths, while role meaning comes
from which role variable you bind that path to.

== Host-side rename of external APFS volume names

You can rename APFS volume names inside external ASIF images directly on the host.

Canonical policy for external sets is:

* `<source-vm> - <original-volume-name>`

Example for all images from `nikopol`:

* `make rename-disk-volumes .tart-vm-name=nikopol`

This defaults to prefix `nikopol - `.
Override the prefix if needed:

* `make rename-disk-volumes .tart-vm-name=nikopol .tart.rename.prefix="legacy - "`

Rename a single image:

* `make rename-disk-volume .tart.rename.image-path="$(HOME)/.tart/disks/nikopol/git-worktree-store.asif"`

Rename explicit attached APFS volumes (Make fan-out, no shell loop in the target body):

* `make rename-disk-attached-volumes .tart.disk.path="$(HOME)/.tart/disks/nikopol/git-bare-store.asif" .tart.disk.volume-devs="disk7s1 disk7s2"`

You can also target one volume directly via pattern target:

* `make disk7s1~rename-volume .tart.disk.path="$(HOME)/.tart/disks/nikopol/git-bare-store.asif"`

Dry-run rename (discover and report candidates, but skip mutating disk updates):

* `make rename-disk-volumes .tart.vm.name=nikopol .dry-run.disk=1`

The rename dry-run mode skips:

* `diskutil mount ...`
* `diskutil unmount ...`
* `diskutil rename ...`

If a disk image does not expose a resolvable APFS volume rename candidate in dry-run,
the image is reported and skipped (non-fatal) so batch dry-run flows continue.

Encrypted/locked APFS candidates are detected from attached metadata and skipped (no marker files are written).

== Dry-run controls (global and scoped)

Dry-run is the default for safety. Use hierarchical dry-run knobs:

* Global: `.dry-run=1`
* Tart-scoped: `.dry-run.tart=1`
* Disk-scoped: `.dry-run.disk=1`
* Packer-scoped: `.dry-run.packer=1`
* APFS ownership-scoped: `.dry-run.apfs.ownership=1`

Live execution requires an explicit override:

* Global live override: `.dry-run=0`
* Scoped live override (example): `.dry-run.disk=0`

Precedence rule:

* scoped value (if set) overrides global `.dry-run`

Examples:

* All supported flows in dry-run mode:
** `make build .dry-run=1`
* All supported flows in live mode:
** `make build .dry-run=0`
* Only Tart operations in dry-run mode:
** `make run-console .dry-run.tart=1`
* Only disk mutating operations in dry-run mode:
** `make rename-disk-volumes .tart.vm.name=nikopol .dry-run.disk=1`

== VSCode-focused background service trimming

During provisioning, `scripts/trim-vscode-vm-services.sh` applies a lightweight background-service profile intended for SSH/VSCode-heavy VM usage.

Default behavior:

* Disable Spotlight indexing across mounted APFS volumes.
* Add `.metadata_never_index` markers on mounted APFS volumes.
* Purge Spotlight assets by default using `mdutil -E` on mounted APFS volumes.
* Fallback to direct `.Spotlight-V100*` cleanup only when `mdutil -E` fails on a writable mount.
* Disable Spotlight daemon:
** `com.apple.metadata.mds`
* Disable media analysis daemons:
** `com.apple.mediaanalysisd`
** `com.apple.photoanalysisd`
* Disable Siri/suggestions style user agents for the primary user:
** `com.apple.assistantd`
** `com.apple.parsecd`
** `com.apple.suggestd`
** `com.apple.knowledge-agent`
* Keep continuity/nearby-sharing enabled by default (safer for multi-host workflows).
* Optionally disable continuity/nearby-sharing style user agents for the primary user when `DISABLE_AIRDROP_CONTINUITY=1`:
** `com.apple.sharingd`
** `com.apple.coreservices.useractivityd`

Safety notes:

* Core SSH/network services are intentionally not disabled.
* Some launchd labels can vary by macOS release; disabling is best-effort.

Control toggles:

* `DISABLE_SPOTLIGHT_INDEXING` (default `1`)
* `SPOTLIGHT_DISABLE_MDS_DAEMON` (default `1`)
* `SPOTLIGHT_PURGE_ASSETS` (default `1`)
* `DISABLE_MEDIA_ANALYSIS_DAEMONS` (default `1`)
* `DISABLE_SIRI_AND_SUGGESTIONS` (default `1`)
* `DISABLE_AIRDROP_CONTINUITY` (default `0`)

Service mode:

* `SERVICE_TRIM_MODE=disable|enable` (default `disable`)

Re-activation examples inside VM:

* Re-enable all service groups currently selected by toggles:
** `sudo env SERVICE_TRIM_MODE=enable /usr/local/sbin/trim-vscode-vm-services`
* Re-enable continuity services only (when they were explicitly disabled):
** `sudo env SERVICE_TRIM_MODE=enable DISABLE_SPOTLIGHT_INDEXING=0 DISABLE_MEDIA_ANALYSIS_DAEMONS=0 DISABLE_SIRI_AND_SUGGESTIONS=0 DISABLE_AIRDROP_CONTINUITY=1 /usr/local/sbin/trim-vscode-vm-services`
